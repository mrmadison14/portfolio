The eval harness that decides whether the bot ships

The setup

We run a multi-agent assistant that answers real customer support questions. It classifies the intent of an inbound message, routes it to a specialist agent, and answers from curated knowledge. Which means a one-line prompt edit is a production change to customer-facing behavior, and there is no compiler, no type checker, and no unit test that will catch a model quietly getting worse at its job.

Early on, the merge criterion was "it looked fine when I tried it in chat." That is not a criterion. That is a mood.

The harness

So the bot ships through an evaluation suite, the same way ordinary code ships through tests. The cases are drawn from real conversations, sanitized, each with an expected outcome. Two tiers keep the loop fast: a smoke suite that runs in minutes during development, and a full suite that gates the merge.

Scores are not one number. Each case is graded on the dimensions that actually matter for a support bot:

  • Routing: did the right specialist handle it?
  • Grounding: does the answer cite the knowledge it used, and is that knowledge real?
  • Abstention: when the bot does not know, does it say so and hand off, instead of improvising?

Abstention deserves the emphasis. A support bot that guesses confidently is worse than no bot, because a wrong answer delivered politely gets believed. Some of our eval cases exist purely to check that the bot declines.

The part nobody warns you about: the judge is noisy

Scoring free-text answers at scale means using a model as the judge, and the judge has moods of its own. The same answer can score an 8 today and a 7 tomorrow. If you treat every dip as a regression you will chase ghosts; if you ignore dips you will ship real breakage.

Triage before touching code

A red case gets rerun before it gets believed. If the score flaps across reruns, it is judge variance and the rubric needs tightening, not the bot. If it holds red, read the actual transcript before changing anything. Only then fix, and reproduce the failure first so the fix has something to turn green.

That red-green discipline is borrowed straight from ordinary test-driven development, and it earns its keep here more than anywhere, because the temptation to "just tweak the prompt and rerun" is enormous and the tweak usually breaks a different case.

Incidents become cases

When the bot misbehaves in production, the conversation gets sanitized and added to the suite as a regression case. The suite is a ratchet: every failure the system has ever taught us is a thing it can no longer do silently.

What I would tell you about it

Treat model behavior like a database schema: never change it without a migration test. The eval suite is what lets a support team ship prompt changes on a Tuesday afternoon without holding their breath. It converts "I think it still works" into "the suite is green, and here is what green means."
